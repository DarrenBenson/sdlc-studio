## What went well

- **`refine add` did its intended job for the first time:** it appended EP0037 to the
  already-decomposed RFC0040 (preserving EP0034), which the last sprint built precisely to unblock.
  The finding -> fix -> use loop closed across three sprints (RETRO0031 flagged it, RETRO0032 built
  it, this sprint used it).
- **The migration is honest about its limits.** It converts what it can deterministically (Effort and
  points on requests/containers) and REPORTS what it cannot (Effort on delivery units - there is no
  honest Effort->Points map, which is the finding the whole model rests on). Dogfooded dry-run on

- **The review approved and its two non-blocking findings were fixed the same sprint:** the point-band
  was duplicated across `refine` and the migration (LL0016), now a single `sdlc_md.size_for_points`
  both share; and a Status-less artefact was over-counted as converted, now reported `needs_manual`.

## What was hard / what stalled

- **The point-band duplication was a self-inflicted LL0016** - I copied `_tshirt_for`'s band into the
  migration instead of extracting it first. Caught in review (I had flagged it in the review prompt
  myself), fixed by moving the band to `sdlc_md`. The lesson holds: extract the shared definition
  BEFORE the second caller, not after.
- Nothing else stalled - a clean 8-point sprint on a well-understood surface.

## Lessons

- Extract a shared definition BEFORE writing the second caller, not after. The point->size band was
  copied into the migration and immediately became an LL0016 drift risk (two paths, one meaning); the
  fix was to move it to `sdlc_md.size_for_points` and have both `refine` and the migration call it.
  A helper worth copying is a helper worth sharing.
- A migration must report what it cannot convert, never guess. Effort predicts cost far worse than
  points, so there is no honest Effort->Points map; the migration writes a Size where the mapping is
  sound (Effort/points on a container) and FLAGS a delivery unit's Effort for human re-sizing rather
  than inventing a number the estimator would then be judged on.
