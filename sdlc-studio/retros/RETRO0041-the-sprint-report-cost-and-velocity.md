## Delivered

- **US0172 (5pt) - per-attempt telemetry.** A record carries an `attempts` list of `{model, tokens}`;
  `attempts_of` reads it, and a legacy record reads as one implicit attempt - non-destructive, so the
  existing evidence needs no migration.
- **US0173 (3pt) - true cost with rework.** `unit_cost` sums cost over attempts, priced offline from a
  `pricing.<model>` config table; an unpriced (or non-positively-priced) model is UNPRICED - its
  tokens counted, its dollars never guessed.
- **US0174 (5pt) - the sprint report.** `sprint_report.py show` composes delivered, spend, velocity,
  accuracy, lessons and tickets - a deterministic page at no model-token cost.
- **US0175 (5pt) - points-per-elapsed-sprint velocity (CR0273).** Primary planning read (ceremony
  included) from the sprint's own run-state; secondary points-per-worker-hour; both descriptive, fed
  to no gate.
- **US0176 (2pt) - rendering gated by config, recording never.** `report.enabled: false` turns the
  page off; telemetry keeps recording regardless.

## What went well

- **Dogfooding caught two real bugs before the review even ran.** Running `accuracy` on RETRO0040
  read 43h of elapsed from a STALE run-state (an old runner run left open), not this sprint - fixed to
  require the run-state's batch to name this sprint's units. And the operator-supplied 1.5h gave 13.3
  points/hour, which validated the operator's own felt velocity heuristic (~1 man-month/hour).
- **The design held the honesty line by construction:** actual spend is a measurement, no avoided-cost
  headline, non-destructive per-attempt schema, rendering-gated-not-recording.

## What was hard / what stalled

- **The independent review found THREE MAJORs in the cost-honesty path - exactly why the estimator
  gets an adversarial pass.** (1) The report crashed on a tokens-only attempt (a `None` model reaching
  a `join`). (2) A negative configured price was honoured and SUBTRACTED from the batch total - a
  flattering figure, the precise class this subsystem exists to prevent. (3) The two honest
  subsystems contradicted: `accuracy` read flat `tokens` while the spend summed `attempts`, so an
  escalation would read UNMEASURED in the ratio while priced in the report. All three fixed with
  regression tests, plus a foreign-model mispricing (substring match) and an open-run-state elapsed
  confounder. Two REQUEST-CHANGES... no - one REQUEST-CHANGES, then APPROVE.

## Lessons

- When two subsystems each claim to be the honest one (here: the estimate-vs-actual ratio and the
  spend report), they must read the SAME source for the same quantity, or they will contradict on
  exactly the interesting case (an escalation). A new richer schema (`attempts`) is a fork in the
  data until EVERY reader is taught to read it; reconcile them in the same change, and test a record
  in the new shape through BOTH readers.
- A configured number is an input, not a fact: a price from config must be range-checked (a negative
  price subtracts, a zero is indistinguishable from unpriced) exactly as a user-supplied size is. The
  estimator's whole discipline is refusing to let an unchecked number wear the authority of a
  measurement - pricing is no exception.
- Dogfooding a measurement feature on real project data finds confounders a fixture never will: the
  stale-run-state elapsed (43h from an unrelated open run) only showed because the live repo HAD an
  old open run-state. Run a new measurement against the real repo before trusting it.
