# RETRO-0028: The sizing-completion sprint: the model works, the input was under-sized, and we proved it

> **Date:** 2026-07-15
> **Batch:** CR0270, CR0268, CR0269
> **Goal:** complete the RFC0038 size model (T-shirts on requests, points on delivery units) and unblock the forward-port
> **Delivered:** 3 / 3   **Blocked:** 0

## Delivered

- CR0269 - size by what a thing IS. A CR and an RFC carry a T-shirt `Size` (S/M/L/XL); a story and a
  bug carry `Points`. The grooming gate is type-aware, with legacy tolerance so existing pointed CRs
  still read. It caught its own author one command after ship, refusing a bug filed with `--size`.
- CR0268 - an epic carries a T-shirt Size AND a `Derived Point Total` reconcile recomputes from its
  stories, so the roll-up cannot silently drift.
- CR0270 - every evidence record is stamped with its project at write time, from the git remote; 400+
  records backfilled value-for-value; `retro collate` computes tokens-per-point per `(project, model)`
  cell and refuses a figure pooled across cells. The forward-port unblocker.

## Blocked / deferred

- Nothing blocked. BG0148 raised: the two creators disagree on a CR's size (file_finding writes a
  T-shirt, artifact.py has no `--size` and writes Points) - the LL0016 follow-up CR0269 flagged.

## What went well

- **A blind decomposition experiment answered the operator's question and vindicated the model.** Two
  independent estimators, blind to cost, decomposed the four points-model CRs into stories: sum of
  story points x the 25,000 seed predicted the batch to 1.00x (887,500 vs 887,704), where the
  single-shot CR points had missed by 0.56x. The estimators agreed exactly on 3 of 4. Decomposition
  does not just make Done checkable - it makes the estimate accurate (LL0038).
- **The gate caught its author instantly.** CR0269 shipped the "a bug carries points, not a T-shirt"
  rule, and refused the very next bug this session filed with `--size`. The rule was live and correct
  within one command.
- Verification by attack held all sprint: the epic roll-up driven through reconcile, the project stamp
  proven on a fresh record, the cross-project pool refused, the creator disagreement found by running
  both creators rather than reading either.

## What was hard / what stalled

- **My single-shot CR sizing is an unreliable input, and it under-sized every unit.** The forecast
  missed at 0.44x because I put casual one-shot points on the CRs at filing (5, 8, 5, 2), and the
  blind experiment showed the honest decomposed sizes were ~1.75x larger. The validated method is
  relative, decomposed, multi-estimator - not a number read off an artefact by one estimator.
- **artifact.py, the canonical creator, cannot T-shirt-size a CR** (BG0148). CR0269 stayed in its lane
  and left it legacy-tolerant, which is correct, but the model is not coherent until it closes.
- The units ran long again (0.32x-0.56x), for the now-understood reason: a repo-wide change (a
  reconcile lane, a 400-record backfill, a doc sweep) does not show in the artefact text a single
  estimator sizes from.

## Lessons

- Decomposition makes the estimate accurate, not only the Done checkable. A blind decomposed estimate
  hit the batch to 1.00x where a whole-unit single-shot guess missed by half. Break a request into the
  stories you would actually create, and the hidden work - and the honest size - surfaces.
- A single-estimator one-shot size read off an artefact is a weak input. The method that predicts is
  relative, decomposed and multi-estimator; it converges (independent estimators agree) and it is
  honest (nobody sees the outcome).
- A cross-cohort figure must be refused, not pooled - now enforced in code (`collate` refuses a rate
  across `(project, model)` cells), so the confound that produced two withdrawn findings this week
  cannot recur through the tooling.

**The forecast missed at 0.44x - and for the first time we know why BEFORE writing this, with proof.**

Forecast 250,000 (10 points x 25,000), actual 564,066. All three units under-forecast (0.40x, 0.32x,
0.56x). But this is not a fourth falsification of the estimator. It is under-SIZING of the input, and a
blind experiment this session established it beyond the anecdote:

CR0268 appears in BOTH this sprint and the decomposition experiment. I sized it **2** at filing. Two
independent blind estimators decomposed it to **6.5**. At 6.5 points it cost **23,679 tokens per
point** - dead on the 25,000 seed. The rate was never wrong; my single-shot points were 1.75x too low.

So the honest reading of this sprint's 0.44x is: **the model is sound, the input was casual.** The fix
is already shipping - CR0269 (this sprint) moves CRs off points onto T-shirts, and CR0271 (next) forces
CRs to decompose into properly-sized stories. The next sprint measured in STORY points, sized by
decomposition, is the real out-of-sample test of the 25,000 rate.
