# RFC-0028: the generated team: project-native personas as the v4 differentiator

> **Status:** Accepted
> **Date:** 2026-07-10
> **Created-by:** sdlc-studio file

## Summary

The README sells living Cooper personas, but the working team is three static cards shipped identically to every project, and nothing generates the stakeholder panel consult expects. Design (operator-directed, research-grounded): analyse the project (PRD/TRD/TSD/repo map/config, mapped onto behavioural variables and risk axes, never demographics) and GENERATE the team - fresh named individuals per project on the amigo-template schema into personas/seats/ per RFC0021's converged model - plus a stakeholder panel (new stakeholder-template: goals, veto lines, evidence read, Cooper Customer/Served designation). Signal-driven squad: 3 core amigos + up to 2 extra seats (Security/SRE/Data/UX on strong signals; cast stays 3-5 per Cooper). Multi-choice questions whenever signals are ambiguous (house Analyse-Present-Enrich-Write idiom); never-clobber authored seats (enrichment diffs only); generated cards labelled provisional-unverified until operator review (Cooper's rule). Offered after PRD and at project upgrade, never auto. Competitive research: BMAD/ChatDev/MetaGPT ship STATIC role casts - the falsifiable README claim gets replaced by the true conjunction (generated-from-your-project + both sides of the table + mechanical independence). Cooper research encodes: one-Primary-per-interface validation, Serves: requirements-coverage tagging (personas that arbitrate, not decorate), anti-sycophancy objection quota, buyer-never-overrides-Primary arbitration, Life goals field, anti-fluff rule.

## Design Options

- **A (chosen): full generation engine as designed - team + stakeholders + Cooper usage pass + RFC0021 completion + positioning, one epic gating v4**
- **B: team generation only, stakeholders and Cooper pass post-v4 - smaller gate, weaker launch story**
- **C: enrich the three static cards per-project without fresh identities - cheapest, rejected by operator (fresh people per project chosen)**

## Recommendation

Option A. Operator directive: fully fleshed out before v4 goes live; design decisions taken via plan-mode Q&A (fresh identities, 3+2 squad, PRD/upgrade triggers); plan approved 2026-07-10.

## Consult Outcome (Three Amigos, plan-review phase, 2026-07-10)

Three seat-resolved reviews, all **APPROVE-with-conditions**; verdicts recorded in
`reviews/plan-review-verdicts.md`. The conditions are folded into the epic slicing below and
are binding ACs, not advisories:

- **Dani (Engineering), blocking:** the stale-amigos precedence trap - `persona_resolve`
  checks `personas/amigos/` first and every upgraded project has generic cards there, so a
  generated team in `seats/` would be silently shadowed forever. Resolution: the converged-home
  work (CR-D scope) is a **stated prerequisite** of team generation; generation detects
  role-claiming `amigos/` cards and migrates/retires or hard-stops (incl. `--dry-run`); the
  upgrade flow offers generation BEFORE, and suppresses, the default-amigo install.
- **Sam (QA), blocking:** the D4 "review render mandatory = hard error" gets an owner - a new
  error-level `validate.py` seats check (role comment, render headings, provenance stamp,
  demographic-token denylist, seat-cast cap); the `Serves:` checker activates only on first use
  (or config opt-in) and requires named personas to resolve to files; eval scenario 07's fixture
  is ambiguous BY DESIGN (conflicting risk classes, absent compliance) with scripted answers and
  the expected question axis in grading_notes.
- **Lena (Product):** the default question flow is hard-capped at <=4 (risk class, extra-seats
  multi-select, stakeholder-cast multi-select, compliance only-if-triggered; `--quick` <=1);
  remaining inferences are defaulted-and-displayed with one batch adjust prompt. The flow closes
  with a batch-accept that clears TEAM seat provisional labels in-flow (stakeholder/design cards
  and headless runs keep them, surfaced via a status advisory and the consult header). Life
  goals generate only on a strategy-tier signal. A standalone repo-map-only path is documented
  (brownfield taster), with a hint after `review generate`.
- **Shared:** never-clobber discriminates authored-vs-generated by provenance stamp PLUS content
  hash (an operator-edited generated card counts as authored); re-runs key on the declared role
  (one generated card per role, diffing its own prior); `check_personas` learns the
  `stakeholders/` schema and the seats/ layout so the generator's canonical output never trips
  the validator; one-Primary ships as a WARNING (error only when two Primaries declare the same
  optional `Interface:`); anti-fluff splits into a mechanical denylist + named judgement in
  `persona review`.

Operator authority: design pre-approved via plan mode 2026-07-10; conditions accepted.
