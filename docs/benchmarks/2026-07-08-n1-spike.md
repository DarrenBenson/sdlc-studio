# N=1 de-risking spike (2026-07-08)

> Per the pre-registered [protocol](protocol.md): this is the N=1 spike that runs before the
> full N=5 measured run, to prove the harness end-to-end and get an early effect-size read.
> Published regardless of outcome, per the anti-hype commitment - and this result is **not
> flattering**.

**Model:** claude-sonnet-5 (both arms, pinned). **Harness:** `tools/bench/runner.py`.
**Fixtures:** `greenfield-csv-dedupe`, `brownfield-pagination`, `brownfield-lru-cache`
(all three in `tools/bench/fixtures/`, hidden suites validated against both a seeded-bug and
a reference-fix version before this run - see their `hidden/test_hidden.py`).

## Headline result

At N=1, on these three fixtures, **arm B (plain Claude Code + a genuinely good CLAUDE.md)
was not beaten by arm A (the sdlc-studio pipeline)** on any of the four pre-registered
metrics:

| Fixture | Arm | Tokens | Wall time (s) | Test-run passes | Defect escape | Rework |
| --- | --- | --- | --- | --- | --- | --- |
| brownfield-pagination | A | 36,345 | 49.4 | 1 | No | No |
| brownfield-pagination | B | 42,070 | 85.2 | 2 | No | No |
| brownfield-lru-cache | A | 42,242 | 52.6 | 1 | No | No |
| brownfield-lru-cache | B | 37,728 | 58.3 | 3 | No | No |
| greenfield-csv-dedupe | A | 48,744 | 123.4 | 2 | No | No |
| greenfield-csv-dedupe | B | 43,079 | 109.6 | 1 | No | No |

- **Defect escapes: 0/3 for both arms.** Every run - both arms, all three fixtures - passed
  its held-back suite on the first declared-done state. No differentiation on the metric
  the protocol names as the primary oracle-measured signal.
- **Rework: 0/3 for both arms** (self-reported). Both arms reported "done in one pass" -
  no run found a real defect in its own solution and had to re-fix it. **This is distinct
  from "test-run passes" above** (recorded in `runs.jsonl` as `iterations`, renamed here for
  clarity): re-reading all six transcripts, every run >1 was a *deliberate mutation-check* -
  the arm re-ran its new test against the original buggy code to prove the test would have
  caught it, then restored the fix - not a fix-forward correction after failure. That is a
  verification-quality practice, not rework, so `rework=false` is accurate for all six runs
  despite the pass-count differing. Worth noting as a mild, non-metric signal: arm A needed
  fewer distinct test-run passes on 2/3 fixtures (pagination, lru-cache).
- **Tokens:** arm A cheaper on 1/3 fixtures (pagination), arm B cheaper on 2/3
  (lru-cache, csv-dedupe).
- **Wall time:** arm A faster on 2/3 fixtures (pagination, lru-cache), arm B faster on 1/3
  (csv-dedupe).

No consistent directional win either way on the four pre-registered metrics, at n=1 per cell.

## Why this result likely doesn't mean what it might look like it means

Reading the six transcripts, all three arm-A runs **explicitly declined to invoke the
sdlc-studio pipeline**, reasoning (in the arms' own words) that a one-file bugfix or small
feature addition didn't warrant a full PRD/epic/story cycle - which is exactly what
`pipeline_CLAUDE.md` tells them to judge for themselves. The practical effect: on these
three fixtures, **arm A behaved like arm B** - both wrote a direct fix plus a regression
test and ran it. The spike did not yet exercise the pipeline's actual differentiating
claims (spec extraction surfacing hidden requirements, drift detection across a longer
task, reconcile-driven status honesty) at all - both arms were, in effect, running the
same "careful engineer with a good CLAUDE.md" strategy this whole time.

This is the spike doing its job: it surfaces a real design gap in the fixture set before
the 5x cost is committed. The task set (three small, well-specified, single-file tickets)
is the wrong shape to detect the pipeline's claimed value, if that value is concentrated in
multi-file/multi-session work or under-specified tickets where spec extraction and drift
control should matter more. Small, clean tickets are exactly where a lightweight,
judgement-driven baseline should hold its own - and it did.

## Other caveats (publishing honestly, not just the favourable ones)

- **n=1 per cell.** No error bars are meaningful yet; every number above could look
  different at N=5. Read this as "no red flags disqualifying the harness," not "the
  pipeline underperforms" - the sample is too small to claim either direction.
- **Arm B's isolation was instruction-level, not environmental.** Both arms ran as
  subagents in the same session that has the sdlc-studio skill loaded; arm B was told not
  to invoke it, not prevented from doing so by removing it from the environment. A
  well-behaved model followed the instruction in all three runs (confirmed by reading each
  transcript), but this is weaker than true removal and worth strengthening before N=5 if
  feasible.
- **Tokens/wall-time were not captured automatically by the harness.** Defect-escape
  scoring is fully automatic and oracle-based (`runner.py score`, the held-back suite -
  never hand-judged). Tokens and wall-time, by contrast, were read by the operator from
  the orchestrating tool's own per-run usage figures (`subagent_tokens`, `duration_ms`)
  and passed to `runner.py record` by hand - consistent across both arms, but not parsed
  from a transcript by the harness itself. CR0178's "captured automatically... no
  hand-scored outcomes" AC is only partly met as a result; closing this gap (an automatic
  transcript parser) is worth doing before N=5 if this harness gets reused at scale.
- **Two hardening gaps found by independent review, fixed same-session (did not affect
  this run's results, confirmed by re-scoring both arms after the fix):** (1)
  `greenfield-csv-dedupe`'s hidden suite invoked `cli.py` via subprocess with
  `cwd=workspace`, so a stray workspace file shadowing a stdlib module name could have
  corrupted scoring undetected - now runs with `PYTHONSAFEPATH=1` so the workspace
  directory never enters `sys.path`. (2) the hidden suite never exercised the
  CRLF-tolerance case the ticket explicitly asks for - added; both arms' solutions still
  pass (7/7, up from 6/6), confirming they handled it correctly all along.

## What this changes before the N=5 run

Recommend **not** proceeding straight to N=5 on this fixture set as-is. Worth deciding
first (a project decision, not this spike's call): either (a) accept that "no measurable
difference on small well-specified tickets" is itself a real, publishable finding and run
N=5 on these fixtures anyway to firm up the estimate, or (b) add a fixture (or revise
`pipeline_CLAUDE.md`'s guidance) sized/shaped to actually exercise multi-file scope,
ambiguous requirements, or a longer task where drift is more likely - the RFC's real target
claim - before spending the full 5x budget on a task set that may not be able to show a
difference either way.
